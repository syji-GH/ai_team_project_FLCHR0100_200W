# 專案規範 — 產品化 Checklist

> 每次開始新功能、新模組、或 PR 前，請對照以下 checklist。
> 標記 🚨 為高風險項目，未完成前不得上線。

---

## 📋 使用方式（給 Claude Code）

當開發者說「開始新功能」或「我要建一個新專案」時，請：
1. 主動詢問以下 Phase 1 的商業驗證項目是否完成
2. 每次生成代碼時，確保 Phase 2–3 的技術項目都有對應實作
3. 若發現缺失項目，在回覆開頭明確標示：「⚠️ 尚未完成：[項目名稱]」
4. 不要跳過任何標記 🚨 的項目

---

## Phase 1｜商業驗證（動工前必須完成）

- [ ] 🚨 定義 North Star Metric，與「節省或創收 200 萬 NTD/年」直接連結
- [ ] 🚨 完成 ROI 計算表：現況成本 → 導入後節省 → 年化金額
- [ ] 訪談 5+ 位目標用戶，確認痛點頻率與嚴重度（有文字記錄）
- [ ] 確認競品與替代方案（Excel/人工），明確切入優勢
- [ ] 🚨 LLM/API 成本估算完成，確保 margin ≥ 60%
- [ ] 有明確的使用者 persona 與使用情境文件

---

## Phase 2｜用戶體驗 & 可進入性

- [ ] 非開發者能在 10 分鐘內開始使用（onboarding flow 已實測）
- [ ] 首次使用到第一個 aha moment ≤ 10 分鐘
- [ ] 錯誤訊息有引導說明（不只說「錯了」，要說「怎麼修」）
- [ ] 有 README 說明安裝、使用、維護方式

---

## Phase 3｜技術品質（每次生成代碼都必須遵守）

### 環境與部署
- [ ] 不依賴開發者本機環境（Docker 或 cloud 可重現）
- [ ] 至少部署到一個可分享的 URL
- [ ] 建立 .env 管理，環境分層（dev / staging / prod）
- [ ] 有版本控制（Git），可納入公司資產長久營運

### 代碼品質
- [ ] 🚨 敏感資訊不在程式碼中（統一使用環境變數）
- [ ] 核心邏輯有基礎單元測試，覆蓋率 ≥ 60%
- [ ] 所有外部 API 呼叫有 retry logic & 友善錯誤訊息
- [ ] 建立 CI/CD pipeline，每次上線有自動化 gate

### 可觀測性
- [ ] Observability 接入（Sentry 等），5 分鐘內能察覺 P0 錯誤
- [ ] 有基本的錯誤處理，不會白屏或崩潰

---

## Phase 4｜安全性 🚨

- [ ] 🚨 Auth 機制完整（JWT/OAuth 2.0），無不安全認證
- [ ] 🚨 敏感資料符合個資法，無未授權第三方傳輸
- [ ] 🚨 有 rate limiting & input validation，防止 abuse 與 prompt injection
- [ ] Database migration 策略明確，新功能不 break 現有用戶資料
- [ ] Feature flag 系統：新功能逐步 rollout

---

## Phase 5｜商業模式（上線前）

- [ ] 選定定價模式（月費/用量計費/授權），對應節省價值的 10–20%
- [ ] 有 free trial 或 freemium，降低新創採用門檻
- [ ] Unit Economics 估算：LTV:CAC ≥ 3:1
- [ ] SLA 條款（uptime ≥ 99.5%）及退款政策草稿

---

## Phase 6｜客戶成功（上線後持續追蹤）

- [ ] 簽下 3–5 家付費 pilot 客戶（≥ 5,000 NTD/月）
- [ ] NPS 追蹤機制建立
- [ ] 定期量化客戶節省成果並記錄
- [ ] 年度 ROI 報告機制（作為續約 & 漲價依據）

---

## 🤖 給 Claude Code 的自動檢查指令

當開發者輸入以下指令時，請執行對應動作：

**`/check-product`** — 輸出目前專案對照完整 checklist 的狀態報告
**`/check-security`** — 掃描當前代碼庫，列出所有 Phase 4 安全項目的狀態
**`/check-roi`** — 詢問開發者 Phase 1 各項目的完成情況，並輸出 ROI 準備度評估
**`/check-deploy`** — 確認 Phase 3 環境與部署項目，輸出上線前 checklist

---

## 代碼生成規範

每次生成代碼時，預設必須包含：

```
1. .env.example 範本（若涉及環境變數）
2. try/catch 錯誤處理（所有 async 函數）
3. input validation（所有用戶輸入）
4. 對應的單元測試骨架
```

若以上任何一項因需求特殊無法包含，請在回覆中明確說明原因。
