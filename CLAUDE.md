# 專案規範 — 產品化 Checklist（Vibe Coding 強化版）

> 每次開始新功能、新模組、或 PR 前，請對照以下 checklist。
> 標記 🚨 為高風險項目，未完成前不得上線。
> 標記 👁️ 為「非技術創辦人可自行驗收」的項目。

---

## ⚠️ Vibe Coding 特別警告（每次生成代碼前必讀）

本專案由非技術創辦人主導，100% 使用 AI 生成代碼。這代表：

1. **每次生成代碼後，必須主動說明這段代碼做了什麼、有什麼風險**
2. **禁止生成「能跑但你不知道在跑什麼」的代碼**
3. **所有涉及金流、個資、認證的代碼，必須附上白話說明與驗收步驟**
4. **若 AI 建議跳過某安全項目「因為比較快」，請明確拒絕並說明風險**
5. **每次 session 結束前，主動輸出本次變更的風險摘要**

---

## 📋 使用方式（給 Claude Code）

當開發者說「開始新功能」或「我要建一個新專案」時，請：
1. 主動詢問以下 Phase 1 的商業驗證項目是否完成
2. 每次生成代碼時，確保 Phase 2–3 的技術項目都有對應實作
3. 若發現缺失項目，在回覆開頭明確標示：「⚠️ 尚未完成：[項目名稱]」
4. 不要跳過任何標記 🚨 的項目
5. 所有 🚨 項目必須附上「非技術人員驗收方式」

## Phase 0｜Vibe Coding 地基（動工前必須完成）🆕

> 這個 Phase 是針對 100% AI 生成代碼的額外保護層。

- [ ] 🚨 👁️ 選定唯一的技術顧問或審查夥伴（每月至少一次代碼審查）
- [ ] 選定技術棧並鎖定（不允許 AI 隨意切換框架），記錄在此文件
- [ ] 🚨 建立 Git 版本控制，每次功能完成必須 commit（不允許只在本機）
- [ ] AI 生成的代碼必須存放在可審計的 repo，不允許「只在本機跑」

**技術棧鎖定記錄（請填寫）：**
```
前端：___________
後端：___________
資料庫：___________
部署平台：___________
金流服務商：___________
```

## Phase 1｜商業驗證（動工前必須完成）

- [ ] 🚨 定義 North Star Metric，與「節省或創收 200 萬 NTD/年」直接連結
- [ ] 🚨 完成 ROI 計算表：現況成本 → 導入後節省 → 年化金額
- [ ] 訪談 5+ 位目標用戶，確認痛點頻率與嚴重度（有文字記錄）
- [ ] 確認競品與替代方案（Excel/人工），明確切入優勢
- [ ] 🚨 LLM/API 成本估算完成，確保 margin ≥ 60%
- [ ] 有明確的使用者 persona 與使用情境文件
- [ ] 🚨 **Kill Switch 條件**：若訪談後痛點頻率 < 每週一次，或 ROI < 50萬/年，暫停開發重新評估

---

## Phase 2｜用戶體驗 & 可進入性

- [ ] 非開發者能在 10 分鐘內開始使用（onboarding flow 已實測）
- [ ] 首次使用到第一個 aha moment ≤ 10 分鐘
- [ ] 錯誤訊息有引導說明（不只說「錯了」，要說「怎麼修」）
- [ ] 有 README 說明安裝、使用、維護方式

---

## Phase 3｜技術品質（每次生成代碼都必須遵守）

### 環境與部署
- [ ] 不依賴開發者本機環境（Docker 或 cloud 可重現）
- [ ] 至少部署到一個可分享的 URL
- [ ] 建立 .env 管理，環境分層（dev / staging / prod）
- [ ] 有版本控制（Git），可納入公司資產長久營運
- [ ] 🆕 部署者交接文件完整（若部署者離職，繼任者能在 1 天內接手）

### 代碼品質
- [ ] 🚨 敏感資訊不在程式碼中（統一使用環境變數）
- [ ] 核心邏輯有基礎單元測試，覆蓋率 ≥ 60%
- [ ] 所有外部 API 呼叫有 retry logic & 友善錯誤訊息
- [ ] 建立 CI/CD pipeline，每次上線有自動化 gate
- [ ] 🆕 API versioning 策略明確（/v1/...），避免未來 breaking change
- [ ] 🆕 所有 API 端點有 request/response schema 文件（OpenAPI/Swagger）

### 可觀測性
- [ ] Observability 接入（Sentry 等），5 分鐘內能察覺 P0 錯誤
- [ ] 有基本的錯誤處理，不會白屏或崩潰
- [ ] 🆕 👁️ 有 dashboard 可讓非技術創辦人即時看到：用量、錯誤率、費用

## Phase 4｜安全性 🚨（Vibe Coding 強化版）

> 以下每項都必須附上「非技術人員的驗收方式」

### 認證與授權
- [ ] 🚨 👁️ Auth 機制完整（使用 Clerk / Supabase Auth / Auth0 等成熟服務，禁止自刻）
- [ ] 🚨 不同角色的用戶只能看到自己的資料（請 AI 生成測試情境並實測）
- [ ] 🚨 登入失敗 5 次後鎖定帳號，防止暴力破解
- [ ] Session 過期機制設定（閒置 30 分鐘自動登出）

### 個資保護（PDPA/個資法）
- [ ] 🚨 👁️ 有隱私權政策頁面，說明收集哪些資料、如何使用
- [ ] 🚨 用戶可以要求刪除自己的資料（刪除流程已實作）
- [ ] 個資不得傳送給未授權第三方（請 AI 列出所有資料流向並確認）
- [ ] 資料保存期限明確，超過期限自動刪除或匿名化

### 金流安全
- [ ] 🚨 👁️ 金流使用 Stripe / 綠界 / 藍新等合規服務，**禁止自己儲存信用卡號**
- [ ] 🚨 所有金流 webhook 有簽名驗證（防止偽造付款通知）
- [ ] 付款記錄完整保存，可供對帳與退款使用
- [ ] 🆕 每月財務對帳機制建立（系統金額 vs. 金流服務商金額需一致）

### 企業資料保護
- [ ] 🚨 多租戶隔離：A 公司的資料絕對無法被 B 公司存取（請 AI 說明隔離機制）
- [ ] 企業資料備份每日執行，可在 4 小時內還原
- [ ] 🆕 資料匯出功能：企業客戶可完整取回自己的資料（避免被綁架）

### 代碼層安全
- [ ] 🚨 所有 DB query 使用 parameterized query，禁止字串拼接（防 SQL injection）
- [ ] 🚨 有 rate limiting & input validation，防止 abuse 與 prompt injection
- [ ] 🆕 Secret scanning 整合進 CI（使用 GitHub secret scanning 或 git-secrets）
- [ ] 🆕 上線前對照 OWASP Top 10 自查清單（請 AI 逐條說明本專案的對應狀態）
- [ ] Database migration 策略明確，新功能不 break 現有用戶資料
- [ ] Feature flag 系統：新功能逐步 rollout

### 👁️ 非技術創辦人的安全驗收清單（每次上線前自行執行）
```
□ 用不存在的帳號登入 → 應顯示錯誤，不得有系統資訊洩漏
□ 用 A 帳號嘗試存取 B 帳號的資料 URL → 應被拒絕
□ 在輸入欄位輸入 <script>alert(1)</script> → 不應彈出任何視窗
□ 用假的付款資訊測試 → 系統應拒絕，不應有任何實際扣款
□ 請 AI 掃描目前 .env 以外的地方是否有 API key 外洩
```

## 代碼生成規範

每次生成代碼時，預設必須包含：

```
1. .env.example 範本（若涉及環境變數）
2. try/catch 錯誤處理（所有 async 函數）
3. input validation（所有用戶輸入）
4. 對應的單元測試骨架
5. 🆕 白話說明：這段代碼做什麼、處理什麼資料、有什麼潛在風險
6. 🆕 若涉及個資或金流，必須附上資料流向說明
```

若以上任何一項因需求特殊無法包含，請在回覆中明確說明原因。

## 🆕 部署架構規範（預設：GitHub → Vercel + Supabase）

> 本專案採用 Vibe Coding 標準架構，無需人工部署介入。
> 待產品方向確定後，請更新本節內容。

### 架構說明
```
代碼管理：GitHub（私有 repo）
前端部署：Vercel（連接 GitHub，push 即自動上線）
後端/資料庫/登入：Supabase（內建 Auth，禁止自刻登入系統）
```

### GitHub 安全設定
- [ ] 🚨 repo 設為 **Private**，未經授權者無法瀏覽代碼
- [ ] 🚨 啟用 GitHub Secret Scanning，自動偵測 API key 外洩
- [ ] 🚨 `.env` 檔案已加入 `.gitignore`，確認從未被 commit 過
- [ ] 保護 main branch：禁止直接 push，需透過 PR

### Vercel 安全設定
- [ ] 🚨 👁️ 環境變數在 Vercel Dashboard 設定，**不透過任何通訊軟體傳送**
- [ ] Production / Preview / Development 環境變數分開設定
- [ ] 👁️ 每季確認 Vercel 團隊成員名單，移除不必要的存取權

### Supabase 安全設定
- [ ] 🚨 啟用 Row Level Security（RLS）：確保 A 用戶無法存取 B 用戶資料
- [ ] 🚨 使用 Supabase 內建 Auth，**禁止自己寫登入系統**
- [ ] anon key 只有最小讀取權限，service role key 絕不放在前端代碼
- [ ] 👁️ 驗收方式：用 A 帳號登入後，嘗試修改 URL 存取 B 帳號資料 → 應被拒絕

### 👁️ 非技術創辦人的部署驗收步驟
```
□ 確認 GitHub repo 是 Private（不是 Public）
□ 在 GitHub 搜尋自己的 API key 關鍵字 → 應該搜不到任何結果
□ Vercel 環境變數頁面確認 key 存在，但值是隱藏的 ****
□ 用兩個不同帳號測試：各自只能看到自己的資料
```
